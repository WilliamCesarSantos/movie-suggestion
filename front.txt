Analisei bem o projeto. Tem um `openapi.yaml` na raiz, JWT, endpoints bem definidos — perfeito para spec driven development. Aqui vai minha sugestão de stack:

---

## Stack sugerida

### **React + TypeScript + Vite**

É a combinação mais adequada para o seu objetivo. Veja o raciocínio:

**React + TypeScript** porque o foco é aprender spec driven development, e TypeScript permite tipar exatamente os contratos que a OpenAPI spec define. Você vai gerar tipos a partir do `openapi.yaml` e o compilador vai garantir que o front respeita o contrato da API.

**Vite** porque é simples de configurar, rápido e não tem fricção de setup — você foca no aprendizado, não em boilerplate.

---

### Bibliotecas essenciais

**Para spec driven development de verdade:**

- **`openapi-typescript`** — gera tipos TypeScript automaticamente a partir do seu `openapi.yaml`. Você nunca escreve um tipo de resposta na mão; tudo vem da spec.
- **`openapi-fetch`** — cliente HTTP tipado que usa esses tipos gerados. Cada chamada é verificada em tempo de compilação contra a spec.

**Para UI:**

- **Tailwind CSS** — sem overhead de configuração de design system. Suficiente para uma demo.
- **shadcn/ui** ou **Radix UI** — componentes acessíveis e sem opinião de estilo, fácil de integrar com Tailwind.

**Para estado/dados:**

- **TanStack Query (React Query)** — gerencia cache, loading e erros das chamadas HTTP de forma elegante. Funciona muito bem com `openapi-fetch`.

---

### Fluxo spec driven com essa stack

```
openapi.yaml  →  openapi-typescript  →  tipos gerados automaticamente
                                              ↓
                                      openapi-fetch (client tipado)
                                              ↓
                                      TanStack Query (hooks de dados)
                                              ↓
                                      Componentes React
```

Toda vez que você atualizar a spec da sua API, roda um comando e os tipos do front são regenerados. Se o front estiver usando um campo que sumiu da spec, o TypeScript avisa na hora.

---

### Estrutura mínima sugerida

```
frontend/
├── openapi.yaml          # cópia ou symlink da spec do backend
├── src/
│   ├── api/
│   │   ├── schema.d.ts   # gerado pelo openapi-typescript
│   │   └── client.ts     # instância do openapi-fetch
│   ├── hooks/            # hooks com TanStack Query
│   └── pages/            # Login, Sugestões, Filmes
```

---

### Por que não Next.js?

Para uma demo local que consome uma API rodando no Docker, Next.js adiciona complexidade (SSR, rotas de servidor) sem benefício. Vite + React é mais direto e deixa o foco no spec driven development.
